# repower — восстановление печати после потери питания для Klipper

Софтовый плагин для Klipper, который во время печати периодически сохраняет
состояние на диск, а после внезапного отключения питания позволяет продолжить
печать примерно с того места, где она остановилась.

Работает без дополнительного железа — на основе периодических снапшотов.
Аппаратный детектор пропадания питания (PLR-модуль) планируется как
опциональное расширение.

> ⚠️ **Дисклеймер.** Восстановление после потери питания на FDM-принтере —
> вещь по своей природе не на 100% надёжная: возможен небольшой шов или сдвиг
> слоя в месте возобновления. Этот плагин делает «как можно лучше», но не
> гарантирует идеальный результат на каждой модели.

## Что требуется

- Печать ведётся **с хоста** через `[virtual_sdcard]` (Mainsail / Fluidd).
- Включён `[force_move]` с `enable_force_move: True` — нужен, чтобы при
  восстановлении задать текущую высоту Z без хоуминга в деталь.
- Ось Z удерживает позицию при выключенном питании (винтовая/leadscrew —
  почти всегда так). На это опирается возврат по высоте.

Заточено под **CoreXY + проба**, но конфиг универсальный — при другой
кинематике достаточно подправить только макрос `REPOWER_RECOVER`.

## Установка (на MKS Pi / Raspberry Pi)

Склонируй репозиторий на хост и запусти установщик — он **всё делает сам**:

```bash
git clone https://github.com/bigtaed-sys/klipper_repower ~/repower
cd ~/repower
chmod +x install.sh
./install.sh
```

Скрипт автономный и идемпотентный — его можно запускать повторно. Он:

- находит Klipper и каталог конфигов (или возьмёт их из `KLIPPER_PATH` /
  `KLIPPER_CONFIG`, если задать вручную);
- слинкует `repower.py` в `klippy/extras/` и `repower.cfg` в каталог конфигов;
- добавит `[include repower.cfg]` в `printer.cfg`, если его там нет;
- добавит `[force_move] / enable_force_move: True`, если он ещё не включён;
- пропишет секцию `[update_manager repower]` в `moonraker.conf` — обновления
  прямо из веб-интерфейса;
- перезапустит сервис Klipper.

Удалить всё, что добавил скрипт (симлинки + перезапуск):

```bash
./install.sh --uninstall
```

> Записи `[include]`, `[force_move]` и `[update_manager]` при удалении
> остаются в конфигах — убери их вручную, если плагин больше не нужен.

## Как пользоваться

1. Печатаешь как обычно. Плагин молча сохраняет состояние раз в
   `save_interval` секунд **только пока идёт печать**.
2. Пропало питание → принтер выключился.
3. После включения в консоли Klipper появится сообщение, что доступна
   recoverable-печать. Проверить вручную:
   ```
   REPOWER_QUERY
   ```
4. Запусти восстановление:
   ```
   REPOWER_RECOVER
   ```
   Принтер прогреется до сохранённых температур, поднимет сопло, отхоумит
   X/Y, вернётся в сохранённую точку и продолжит печать с нужной строки
   файла.

   Параметры (необязательно):
   ```
   REPOWER_RECOVER Z_HOP=5 TRAVEL_SPEED=150 PRIME=2
   ```
   - `Z_HOP` — подъём по Z перед перемещением (мм, по умолчанию 5)
   - `TRAVEL_SPEED` — скорость перемещения XY (мм/с, по умолчанию 150)
   - `PRIME` — сколько прутка подать перед продолжением, чтобы
     компенсировать вытекание во время простоя (мм, по умолчанию 0)

5. Если возобновлять не нужно — сбрось состояние:
   ```
   REPOWER_CLEAR
   ```

## Команды

| Команда | Назначение |
| --- | --- |
| `REPOWER_QUERY` | Показать, есть ли сохранённое состояние, и его детали |
| `REPOWER_RECOVER` | Полная процедура: прогрев → хоум XY → возврат → продолжение |
| `REPOWER_RESUME` | Низкоуровневая: открыть файл, перемотать на offset, продолжить (вызывается из `REPOWER_RECOVER`) |
| `REPOWER_CLEAR` | Удалить сохранённое состояние |

## Параметры секции `[repower]`

| Параметр | По умолчанию | Описание |
| --- | --- | --- |
| `state_path` | `~/printer_data/repower_state.json` | Куда писать снапшот |
| `save_interval` | `1.0` | Частота снапшотов (сек). Меньше — меньше потерь, но чаще запись на диск |
| `min_z_change` | `0.0` | Снимать снапшот только при изменении Z на N мм (≈ раз в слой), чтобы беречь ресурс SD/flash. `0` — каждый интервал |

## Что сохраняется в снапшоте

- позиция в G-code файле (байтовый offset в `virtual_sdcard`);
- координаты X/Y/Z/E и gcode-координаты;
- целевые температуры сопла и стола;
- обороты вентилятора;
- factors скорости/потока (M220/M221);
- режимы координат и экструзии (G90/G91, M82/M83);
- активный профиль сетки стола (bed mesh) и gcode-offset (Z-offset пробы /
  babystep).

При возобновлении `REPOWER_RESUME` восстанавливает режимы/factors/вентилятор,
**заново загружает сетку стола** (`BED_MESH_PROFILE LOAD=...`) и **применяет
gcode-offset** (`SET_GCODE_OFFSET ... MOVE=0`) — без этого после `G28` теряется
Z-компенсация и печать «всплывает»/смещается. Затем задаёт origin экструдера
(`G92 E`) и продолжает файл ровно с сохранённого байта.

> 🛟 Установщик вставляет свои секции в `printer.cfg` **перед** авто-блоком
> `#*# SAVE_CONFIG`, поэтому калибровки BLTouch и сетки стола не затираются.

## Рекомендуется

Добавь `REPOWER_CLEAR` в свои макросы `PRINT_END` и `CANCEL_PRINT`, чтобы
после штатного завершения не оставалось «протухшего» состояния (плагин и сам
чистит его при нормальном завершении, но явный вызов — мгновенный и
надёжный). Примеры — в комментариях `repower.cfg`.

## Как это работает (коротко)

`repower.py` — модуль Klipper (`klippy/extras/`). По таймеру он опрашивает
`virtual_sdcard`, `toolhead`, `gcode_move`, нагреватели и вентилятор, и
атомарно (через temp-файл + rename, чтобы не словить битый файл при сбое)
пишет JSON-снапшот. На старте Klipper читает снапшот: если печать была
прервана — поднимает флаг `printer.repower.recoverable`. Возобновление
переоткрывает файл, делает `seek()` на сохранённый offset и запускает
`do_resume()` у `virtual_sdcard`.

## Ограничения и планы

- Между последним снапшотом и моментом отключения теряется до `save_interval`
  секунд печати → в этом месте возможен небольшой шов.
- Предполагается, что Z не сдвинулся при выключении.
- **Планируется:** аппаратный детектор питания (GPIO + конденсатор) для
  сохранения точного состояния в момент сбоя и аварийной парковки.
