# repower — восстановление печати после потери питания для Klipper

Софтовый плагин для Klipper, который во время печати периодически сохраняет
состояние на диск, а после внезапного отключения питания позволяет продолжить
печать примерно с того места, где она остановилась.

Работает без дополнительного железа — на основе периодических снапшотов.
Аппаратный детектор пропадания питания (PLR-модуль) планируется как
опциональное расширение.

> ⚠️ **Дисклеймер.** Восстановление после потери питания на FDM-принтере —
> вещь по своей природе не на 100% надёжная: возможен небольшой шов или сдвиг
> слоя в месте возобновления. Этот плагин делает «как можно лучше», но не
> гарантирует идеальный результат на каждой модели.

## Что требуется

- Печать ведётся **с хоста** через `[virtual_sdcard]` (Mainsail / Fluidd).
- Включён `[force_move]` с `enable_force_move: True` — нужен, чтобы при
  восстановлении задать текущую высоту Z без хоуминга в деталь.
- Ось Z удерживает позицию при выключенном питании (винтовая/leadscrew —
  почти всегда так). На это опирается возврат по высоте.

Заточено под **CoreXY + проба**, но конфиг универсальный — при другой
кинематике достаточно подправить только макрос `REPOWER_RECOVER`.

## Установка (на MKS Pi / Raspberry Pi)

Склонируй репозиторий на хост и запусти установщик — он **всё делает сам**:

```bash
git clone https://github.com/bigtaed-sys/klipper_repower ~/repower
cd ~/repower
chmod +x install.sh
./install.sh
```

Установщик **интерактивный** (меню на whiptail, fallback на простые вопросы).
При первом запуске он проведёт через настройку:

1. **Язык** меню и панели (RU/EN);
2. **Интервал снапшотов** и **длину прочистки**;
3. **Уведомления** — Telegram / ntfy / выключено (+ тест-отправка).

И сделает всё остальное автоматически:

- находит Klipper и каталог конфигов (или из `KLIPPER_PATH` / `KLIPPER_CONFIG`);
- слинкует `repower.py` в `klippy/extras/`;
- установит **копию** `repower.cfg` в каталог конфигов (твои настройки не
  перетираются при `git pull`) и впишет в неё выбранные значения;
- добавит `[include repower.cfg]` и `[force_move] enable_force_move: True`
  **перед** блоком `#*# SAVE_CONFIG` (калибровки целы);
- пропишет `[update_manager repower]` в `moonraker.conf` — обновления из UI;
- перезапустит Klipper.

Полезные режимы:

```bash
./install.sh --reconfigure      # заново пройти настройку (язык/интервал/уведомления)
./install.sh --non-interactive  # без вопросов, со значениями по умолчанию
./install.sh --uninstall        # снять симлинк модуля и перезапустить
```

> При обновлении через Moonraker скрипт запускается **без TTY** и автоматически
> пропускает все вопросы — просто чинит установку, настройки сохраняются.

## Как пользоваться

1. Печатаешь как обычно. Плагин молча сохраняет состояние раз в
   `save_interval` секунд **только пока идёт печать**.
2. Пропало питание → принтер выключился.
3. После включения в консоли Klipper появится сообщение, что доступна
   recoverable-печать. Проверить вручную:
   ```
   REPOWER_QUERY
   ```
4. Запусти восстановление:
   ```
   REPOWER_RECOVER
   ```
   Принтер прогреется до сохранённых температур, поднимет сопло, отхоумит
   X/Y, вернётся в сохранённую точку и продолжит печать с нужной строки
   файла.

   **Важно про каплю:** сопло греется до рабочей температуры **только после**
   подъёма Z и хоуминга X/Y. Поэтому капля пластика при нагреве падает на углу
   (куда встало сопло после `G28`), а не на модель.

   Параметры можно задать один раз как переменные макроса в `repower.cfg`
   (`variable_purge: 8` и т.д.) или переопределить в команде:
   ```
   REPOWER_RECOVER PURGE=12 PRIME=1
   ```
   - `Z_HOP` — подъём по Z перед перемещением (мм, по умолчанию 5)
   - `TRAVEL_SPEED` — скорость перемещения XY (мм/с, по умолчанию 150)
   - `PARK_X` / `PARK_Y` — отъехать сюда для нагрева/прочистки; `<0` —
     выключено, греть на углу после `G28` (по умолчанию выключено)
   - `PURGE` — сколько прутка прочистить после нагрева (мм, по умолчанию 8)
   - `PURGE_RETRACT` — retract после прочистки, чтобы не тянуть соплю при
     возврате (мм, по умолчанию 0.8)
   - `PRIME` — доп. подача прутка после возврата к детали (мм, по умолчанию 0)

5. Если возобновлять не нужно — сбрось состояние:
   ```
   REPOWER_CLEAR
   ```

### Кнопка / диалог в Mainsail и Fluidd

Не нужно править исходники интерфейса. Используются штатные механизмы:

- **Кнопки макросов.** `REPOWER_RECOVER` и др. автоматически появляются в
  панели Macros — их можно закрепить на дашборде и покрасить в настройках UI.
- **Всплывающий диалог.** После сбоя при загрузке Klipper автоматически
  поднимается окно «Power-loss recovery» с кнопками **Recover** / **Discard**
  (через `action:prompt_*` — поддерживается и Mainsail, и Fluidd, без секции
  `[respond]`).

  > ⚠️ **Fluidd** показывает диалоги только «вживую» и не восстанавливает их
  > из истории. Поэтому плагин пере-показывает окно несколько раз после
  > загрузки (`prompt_retries` × `prompt_interval`), чтобы поймать момент, когда
  > ты откроешь интерфейс. Кнопка **Close** останавливает повторы (состояние
  > при этом сохраняется — восстановить можно кнопкой макроса `REPOWER_RECOVER`
  > или снова вызвать диалог через `REPOWER_PROMPT`).

### Нативная панель (форк Fluidd с plugin-UI API)

Если используется форк Fluidd с поддержкой plugin-UI API, плагин отдаёт
**нативную панель «Power Recovery»** (через ключ `fluidd_ui` в `get_status()`).
Панель сделана на вкладках:

- **Вкладка «Status»** — баннер-статус, **прогресс-бар** печати, имя файла,
  **круговые гейджи** температур сопла/стола, высота Z, обдув, координаты X/Y,
  и кнопки **Recover** / **Discard** (с подтверждением) + «показать диалог».
  Блоки видны только когда есть что восстанавливать.
- **Вкладка «Settings»** — поля Purge / Prime / Z hop / Travel / Park X/Y
  (привязаны к переменным макроса `REPOWER_RECOVER`, правишь прямо из UI) и
  **переключатель языка**.

Фишки форка, которые задействованы:

- **Бейдж в меню** — точка-внимание у пункта панели, когда есть прерванная
  печать (`badge`);
- **Карточка на дашборде** — панель доступна и как dashboard-карточка
  (`surface: both`);
- **Живое переключение языка** — выбор `en/ru` в Settings применяется
  **мгновенно** (форк live-перерисовывает манифест), без рестарта. Стартовый
  язык задаётся `language: en|ru` в `[repower]`. Диалоги тоже локализованы.

Панель появляется автоматически при подключении / `FIRMWARE_RESTART`. Стоковый
Fluidd и Mainsail игнорируют ключ `fluidd_ui` — совместимость не ломается.
Отключить панель: `fluidd_panel: False`.

### Push-уведомления

Когда после сбоя при загрузке обнаруживается восстановимая печать, плагин
может прислать уведомление (отправка в фоне, не блокирует Klipper):

- **Telegram** — создай бота через @BotFather, возьми токен и свой `chat_id`:
  ```ini
  [repower]
  notify: telegram
  notify_telegram_token: 123456:ABC-DEF...
  notify_telegram_chat: 111111111
  ```
- **ntfy** — подпишись на тему в приложении ntfy и укажи её:
  ```ini
  [repower]
  notify: ntfy
  notify_ntfy_topic: my-secret-printer-topic
  #notify_ntfy_url: https://ntfy.sh      # для self-hosted
  ```

Проверить настройку: команда `REPOWER_NOTIFY_TEST`. Установщик тоже умеет
отправить тест в конце настройки.

### Восстановление Z через пробу

Чтобы не «доверять» тому, что Z удержал высоту при выключении, плагин может
**измерить истинный Z пробой по чистому участку стола** (никогда не над
моделью). На восстановлении (внутри `REPOWER_RECOVER`):

1. сопло **поднимается** (винтовой Z держит позицию — подъём безопасен);
2. хоумятся X/Y;
3. плагин выбирает **чистую точку** и едет туда: **авто** — рядом с
   bounding box модели (его границы плагин копит во время печати) с зазором
   `recovery_clearance`; если места нет — **фикс. точка** `recovery_probe_x/y`;
   если и её нет/стол занят — `probe_ok=false` и используется **доверие Z**;
4. `PROBE` по чистому столу → re-референс Z по `z_offset` пробы. Так ловится
   даже **просадка Z**, а не только мелкий дрейф;
5. дальше — нагрев, прочистка, возврат к детали, продолжение.

Проба **никогда не опускается над моделью**. Управление: переменная
`variable_use_probe` у `REPOWER_RECOVER` (`1`/`0`) или `REPOWER_RECOVER
USE_PROBE=0`. В панели в строке «Z recovery» видно, что будет применено —
`probe` или `trust`.

> Требуется проба (BLTouch/индуктивный). Подход рассчитан на **винтовой Z**
> (самотормозящий). Для ремённого Z подъём после сбоя менее предсказуем.

## Команды

| Команда | Назначение |
| --- | --- |
| `REPOWER_QUERY` | Показать, есть ли сохранённое состояние, и его детали |
| `REPOWER_PROMPT` | Показать диалог восстановления в Mainsail/Fluidd |
| `REPOWER_RECOVER` | Полная процедура: прогрев → хоум XY → возврат → продолжение |
| `REPOWER_RESUME` | Низкоуровневая: открыть файл, перемотать на offset, продолжить (вызывается из `REPOWER_RECOVER`) |
| `REPOWER_CLEAR` | Удалить сохранённое состояние |
| `REPOWER_SET_LANGUAGE LANG=ru` | Переключить язык панели/диалогов на лету |
| `REPOWER_NOTIFY_TEST` | Отправить тестовое push-уведомление |

## Параметры секции `[repower]`

| Параметр | По умолчанию | Описание |
| --- | --- | --- |
| `state_path` | `~/printer_data/repower_state.json` | Куда писать снапшот |
| `save_interval` | `1.0` | Частота снапшотов (сек). Меньше — меньше потерь, но чаще запись на диск |
| `min_z_change` | `0.0` | Снимать снапшот только при изменении Z на N мм (≈ раз в слой), чтобы беречь ресурс SD/flash. `0` — каждый интервал |
| `prompt_on_startup` | `True` | Автоматически показывать диалог восстановления при загрузке |
| `prompt_retries` | `6` | Сколько раз пере-показать диалог (Fluidd рисует только «живые» ответы — ловим момент подключения браузера) |
| `prompt_interval` | `20.0` | Интервал между повторными показами диалога (сек) |
| `fluidd_panel` | `True` | Отдавать нативную панель в Fluidd (форк с plugin-UI API). Стоковый Fluidd/Mainsail просто игнорирует ключ |
| `language` | `en` | Язык панели и диалогов: `en` или `ru` (применяется после рестарта) |
| `notify` | `none` | Канал push-уведомления о сбое: `none` / `telegram` / `ntfy` |
| `notify_telegram_token` | — | Токен бота Telegram (от @BotFather) |
| `notify_telegram_chat` | — | Telegram chat id (числовой) |
| `notify_ntfy_topic` | — | Тема ntfy (подписка в приложении ntfy) |
| `notify_ntfy_url` | `https://ntfy.sh` | URL сервера ntfy (для self-hosted) |
| `recovery_clearance` | `15` | Зазор (мм) для авто-точки пробы рядом с моделью |
| `recovery_probe_x` | `-1` | Фикс. точка пробы X (запас, если авто не выходит). `<0` — выкл |
| `recovery_probe_y` | `-1` | Фикс. точка пробы Y |

## Что сохраняется в снапшоте

- позиция в G-code файле (байтовый offset в `virtual_sdcard`);
- координаты X/Y/Z/E и gcode-координаты;
- целевые температуры сопла и стола;
- обороты вентилятора;
- factors скорости/потока (M220/M221);
- режимы координат и экструзии (G90/G91, M82/M83);
- активный профиль сетки стола (bed mesh) и gcode-offset (Z-offset пробы /
  babystep).

При возобновлении `REPOWER_RESUME` восстанавливает режимы/factors/вентилятор,
**заново загружает сетку стола** (`BED_MESH_PROFILE LOAD=...`) и **применяет
gcode-offset** (`SET_GCODE_OFFSET ... MOVE=0`) — без этого после `G28` теряется
Z-компенсация и печать «всплывает»/смещается. Затем задаёт origin экструдера
(`G92 E`) и продолжает файл ровно с сохранённого байта.

> 🛟 Установщик вставляет свои секции в `printer.cfg` **перед** авто-блоком
> `#*# SAVE_CONFIG`, поэтому калибровки BLTouch и сетки стола не затираются.

## Рекомендуется

Добавь `REPOWER_CLEAR` в свои макросы `PRINT_END` и `CANCEL_PRINT`, чтобы
после штатного завершения не оставалось «протухшего» состояния (плагин и сам
чистит его при нормальном завершении, но явный вызов — мгновенный и
надёжный). Примеры — в комментариях `repower.cfg`.

## Как это работает (коротко)

`repower.py` — модуль Klipper (`klippy/extras/`). По таймеру он опрашивает
`virtual_sdcard`, `toolhead`, `gcode_move`, нагреватели и вентилятор, и
атомарно (через temp-файл + rename, чтобы не словить битый файл при сбое)
пишет JSON-снапшот. На старте Klipper читает снапшот: если печать была
прервана — поднимает флаг `printer.repower.recoverable`. Возобновление
переоткрывает файл, делает `seek()` на сохранённый offset и запускает
`do_resume()` у `virtual_sdcard`.

## Ограничения и планы

- Между последним снапшотом и моментом отключения теряется до `save_interval`
  секунд печати → в этом месте возможен небольшой шов.
- Возврат по Z: при `use_probe` истинная высота **измеряется пробой** (ловит
  и просадку Z); без пробы — предполагается, что Z удержал позицию.
- **Планируется:** аппаратный детектор питания (GPIO + конденсатор) для
  сохранения точного состояния в момент сбоя и аварийной парковки.
